# PeerChat 项目需求文档

## 1. 项目概述

**PeerChat** 是一个基于浏览器、工作在局域网内的点对点（P2P）聊天工具。它通过极简的 Python 信令服务器交换 WebRTC 连接信息，所有消息、文件和图片均通过 P2P 数据通道直接传输，不经过服务器。界面设计参考 Slack / Notion 风格，使用 Tailwind CSS 实现响应式布局，整体风格扁平、现代、优雅。

## 2. 技术栈

| 层级 | 技术 |
|------|------|
| 前端框架 | 原生 HTML/CSS/JS 或轻量框架（如 Lit / Preact），但必须支持 WebRTC API，推荐直接用 Vanilla JS 模块化组织 |
| CSS 框架 | Tailwind CSS（响应式设计） |
| P2P 技术 | WebRTC (RTCPeerConnection + RTCDataChannel) |
| 信令协议 | WebSocket（JSON 格式） |
| 信令服务器 | Python，使用轻量异步库（如 `aiohttp` 或 `websockets`），提供 WebSocket + 静态文件服务，不需要任何数据库，所有状态存内存 |
| 文件传输 | 通过 WebRTC DataChannel 分片发送，前端实现文件分块与组装，无断点续传 |

## 3. 信令服务器需求

### 3.1 核心职责

- 生成唯一的 8 位数字会话码
- 管理房间（每个房间最多 2 个客户端）
- 在房间内的两个客户端之间转发 WebRTC 信令（offer / answer / ICE candidate）
- 通知对等方连接状态变化（加入、离开）
- 处理重复的会话码生成（保证唯一性，简单方案：随机生成 8 位数字，检查内存中是否已存在）

### 3.2 通信协议（WebSocket JSON）

客户端与服务器建立 WebSocket 连接后，所有消息均采用 JSON 格式，包含 `type` 字段标识动作：

**客户端 → 服务器：**

```json
// 创建新会话
{
  "type": "create_room",
  "nickname": "我的昵称"
}

// 加入已有会话
{
  "type": "join_room",
  "room_code": "12345678",
  "nickname": "我的昵称"
}

// 转发 WebRTC 信令给对等方
{
  "type": "signal",
  "room_code": "12345678",
  "data": { ... }  // 包含 sdp 或 candidate
}
```

**服务器 → 客户端：**

```json
// 创建房间成功，返回房间码
{
  "type": "room_created",
  "room_code": "12345678"
}

// 有对等方加入房间
{
  "type": "peer_joined",
  "peer_nickname": "对方昵称"
}

// 转发对等方的 WebRTC 信令
{
  "type": "signal",
  "data": { ... }
}

// 对等方离开（断开连接）
{
  "type": "peer_left"
}

// 错误信息
{
  "type": "error",
  "message": "房间码不存在 / 房间已满等"
}
```

**信令流程：**

1. 用户 A 点击“创建会话”，服务器生成 `room_code`，返回给 A。
2. 用户 B 通过链接或手动输入 `room_code` 加入，服务器通知 A：“对方已加入”，同时将 B 的昵称发给 A；通知 B：“已加入房间”，并发送 A 的昵称。
3. 客户端（谁先收到 `peer_joined` 谁发起 Offer）按标准 WebRTC 流程交换 SDP 和 ICE candidate，通过服务器 `signal` 消息转发。
4. DataChannel 建立成功后，双方直接开始 P2P 聊天，之后无需服务器中转消息。
5. 任一客户端断开 WebSocket，服务器通知对方 `peer_left`，客户端 UI 回到“等待连接”状态。

### 3.3 极简实现建议

推荐使用 Python 的 `websockets` 库（无需额外框架），服务器同时提供静态文件托管（将前端打包文件夹部署在同一端口）。保持代码行数最少，只包含：

- WebSocket 连接管理
- 房间字典（room_code → { clients: [ws1, ws2], nicknames: ["A", "B"] }）
- 生成唯一 8 位数字函数
- 消息路由（create/join/signal/disconnect）

## 4. 前端功能需求

### 4.1 入口页面（登录/加入）

当用户访问网站时，如果没有携带 `?code=` 参数，展示：

- 顶部居中 PeerChat Logo（SVG）和名称
- 昵称输入框（默认值可为空，提示“输入你的昵称”）
- 两个操作区：
  - **创建新会话** 按钮：点击后调用信令创建房间，成功后跳转至聊天界面，顶部显示会话码（8位数字）。
  - **加入会话** 输入框 + 8 位数字输入（最多8位，仅数字） + “加入”按钮。或者直接提供一个大的输入框让用户输入会话码。
- 如果通过分享链接进入（`?code=12345678`），则：
  - 昵称输入框正常显示（用户可修改）
  - 下方会话码输入框自动填写为链接中的 code，并置于只读状态（或允许修改但预填），用户只需点击“加入”按钮。
  - 加入成功后进入聊天界面。

### 4.2 聊天界面

**布局：**

- 顶部栏（Header）：
  - 左侧：方块P字母+ “PeerChat” 文字
  - 中间偏左：会话码显示（如 “会话码: 12345678”），hover提示点击复制，点击后复制到剪贴板。若已连接对方，则下右侧显示对方昵称（如 “12345678 · 小明”）。未连接时显示“等待对方连接...”。
  - 右侧：连接状态指示灯（圆点）+ 文字（灰色断联 / 橘色等待连接 / 绿色已连接）+ 分享按钮） + 退出按钮
- 主内容区：
  - 消息列表：滚动区域，展示聊天消息。新消息自动滚动到底部（若用户手动上翻则不强制滚底，可添加“新消息”提示条）。
  - 每条消息气泡：
    - 文字消息：显示发送者昵称（首次连续消息可只显示一次）、时间、消息内容。自己的消息靠右，对方靠左。参考 Slack 风格。
    - 图片消息：如果收到的是图片（通过 DataChannel 发送二进制数据，前端判断 MIME 类型），以缩略图或直接以合适宽度展示，点击可放大查看（可用模态框）。
    - 文件消息：显示文件名、大小和下载按钮（点击触发浏览器下载，通过 URL.createObjectURL 或直接保存接收到的文件 Blob）。
- 底部输入区域（固定在底部）：
  - 文本输入框（支持多行，可使用 `textarea` 自适高度）
  - 发送按钮（或 Enter 发送，Shift+Enter 换行）
  - 文件/图片附加按钮（图标使用 SVG）：点击后打开文件选择器，支持图片和任意文件。可设置 `accept="image/*,.pdf,.docx,..."` 但不强制，允许所有文件。
  - 发送文件时，通过 DataChannel 分块传输，界面显示上传进度。

**连接状态指示器（右侧）：**

- 灰色圆点 + “未连接”：WebSocket 未连接或已断开，信令不可达。
- 橘色圆点 + “等待连接”：已创建或加入房间，但对方尚未建立 P2P 连接（等待对方加入或 P2P 握手尚未完成）。
- 绿色圆点 + “已连接”：DataChannel 打开，可以正常聊天和发送文件。

**分享按钮：**

- 仅在创建房间者一侧显示（也可以双方都显示，但会话码在顶部已经可见，分享按钮用于生成完整链接）。
- 点击后，使用 `navigator.clipboard.writeText` 复制链接 `http://当前域名?code=12345678`，并给出“链接已复制”提示（toast）。

### 4.3 离开确认

用户关闭标签页或刷新时，通过 `beforeunload` 事件弹出浏览器默认确认框：“你确定要离开吗？当前会话可能会断开。” 避免误操作。

### 4.4 断开重连

- 如果对方主动断开（收到 `peer_left` 信令），当前客户端恢复到“等待连接”状态（橘色），顶部显示“对方已断开，等待重新连接…”，会话码不变。
- 此时若对方通过相同会话码重新加入（例如分享链接），可以重新建立连接。
- 若己方意外刷新页面，WebSocket 断开，需要重新连接并重新加入同一个房间码（可通过 URL 参数记忆？）由于不持久化聊天记录，刷新后对话清空，但可在加入时重新绑定房间。如果用户刷新，就相当于是新用户了（需要重新输入昵称？）。设计要求：会话码应该保持在 URL 里吗？需求中说明“进入后地址栏不再显示会话码，方便刷新”。这意味着进入聊天界面后 URL 是干净的（不带 `?code`）。但这样刷新后就会丢失会话码，只能重新通过分享链接进入。可以这样设计：一旦进入聊天界面，会话码就显示在界面顶部，用户刷新后，页面变为无 code 的入口页，需要重新输入或由用户复制顶部的会话码去加入。这符合需求中的“方便刷新”指用户刷新时不至于因为 URL 带参数而重复加入。所以我们保持 URL 干净，让用户手动处理刷新（如果刷新，就要重新加入）。在连接状态说明中可以提及。

### 4.5 文件传输规范

- 文件大小限制：理论上可支持较大文件，但建议前端对单文件超过 10GB 时给出警告（提示文件过大可能传输缓慢或失败），不做硬性限制。
- 分块大小：建议 16KB 或 64KB，根据 DataChannel 缓冲区动态调整。
- 传输协议：通过 DataChannel 发送自定义头部（JSON）标识文件元数据，随后跟二进制块。
  - 头部示例：`{ type: "file", name: "report.pdf", size: 1048576, mimeType: "application/pdf" }`
  - 随后发送文件数据块。
- 接收端检测头部后，收集数据块拼接成 Blob，然后生成 URL 用于下载或展示。
- 图片消息同理，头部 `type: "image"`，接收端可将 Blob 显示为图片。

**注意：** DataChannel 支持字符串和二进制。可以专门用一个 DataChannel 传文件，另一个传文本消息，或者复用同一个，通过消息前缀或类型字段区分。简单方案：统一用 JSON 字符串（文本消息），遇到文件/图片则发送一条 JSON 消息告知随后有二进制数据，然后发送二进制。但 DataChannel 的 `send()` 方法可以混用，但需要接收方区分。更好的方式是创建两条 DataChannel：一条用于文本（`chat`），一条用于文件（`file`）。采用“一条通道”且用类型区分也可，但解析会复杂些。我们可以定义：所有数据通过一个 DataChannel 发送，但文本消息直接是 JSON 字符串，文件消息先发送一个 JSON 控制消息（含 type: "file-meta"），然后发送 ArrayBuffer 数据块。接收端通过 `message.data` 的类型进行分流（字符串 vs ArrayBuffer）。

### 4.6 消息展示增强

- 时间戳格式化：显示相对时间（如“刚刚”、“3分钟前”）或具体时间，参考 Slack。
- 当发送图片时，对方直接在消息框内看到图片缩略图，点击放大。
- 当发送文件时，显示文件卡片（文件名、类型图标、大小），并带有下载按钮。
- 系统消息：显示“对方已加入”、“对方已断开”等以灰色文字居中展示。

## 5. UI/UX 设计指南

- 整体配色：浅色背景，主色调选用柔和蓝或经典中性灰，参考 Slack 的 #4A154B 风格或 Notion 的简约白/灰。可使用 Tailwind 默认调色板。
- Logo：扁平风格 SVG，图形结合字母“P”和聊天气泡元素，简洁有力。不要使用 emoji。颜色与主色调匹配。
- 字体：使用系统默认字体栈（Inter, -apple-system, sans-serif）。
- 响应式：移动端布局下，会话码和状态可以换行显示，消息气泡边距缩小，输入区域高度适配。但无需特别复杂的移动端导航，保持可用即可。
- 动画：消息出现时淡入效果，连接状态切换颜色过渡自然。

## 6. 非功能性需求

- **安全性**：仅局域网使用，不涉及加密与认证，信令服务器不暴露公网。但 WebRTC 本身支持 DTLS 加密，无需额外配置。
- **性能**：大文件传输不阻塞 UI，使用 Web Worker 或分片异步发送。
- **浏览器兼容性**：支持现代浏览器（Chrome, Firefox, Edge, Safari），均需支持 WebRTC 相关 API。
